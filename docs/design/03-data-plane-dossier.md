## Prior art for versioned/branchable application data

**Dolt / DoltgreSQL** — MySQL/Postgres-compatible SQL databases where every table is stored as a Prolly Tree (a B-tree variant where node boundaries are content-defined, like a rolling hash) so each version is content-addressed and diffable at row granularity; branch/merge/diff/log map directly onto git verbs, and it literally stores its data in git-style DoltHub repos. Maturity: Dolt is production-stable (MySQL-compatible, ~1.1x slower than vanilla MySQL on sysbench); DoltgreSQL only reached Beta in 2025 and is less battle-tested. Build cost to reimplement: high — a correct Prolly-tree engine plus a full SQL layer is a multi-year project, not something to bolt onto SQLite. Realistic adoption: none directly, but the *idea* (content-defined chunking of structured data for diffability) is worth stealing conceptually. [dolthub/dolt](https://github.com/dolthub/dolt), [DoltHub blog: How to Version Control a Database](https://www.dolthub.com/blog/2026-01-07-how-to-version-control-a-database/), [dolthub/doltgresql](https://github.com/dolthub/doltgresql)

**Neon** — Separates Postgres compute from a custom distributed storage layer (Pageserver) that never overwrites pages in place; every branch is a metadata pointer into the parent's WAL/page history, and new pages are written only on divergence. Branching a multi-TB database takes under a second because nothing is copied up front. Maturity: production, widely used for preview-environment/CI branching. Build cost: very high to replicate generally (it requires owning the storage engine underneath Postgres), but the *narrow trick* — CoW at the file/page level plus a WAL replay boundary — is exactly what's cheap to approximate on SQLite, since SQLite's file format is a single b-tree file and OS-level CoW (APFS/ZFS clonefile, or `cp --reflink`) gets you 90% of this for free without touching WAL internals. [Neon: Instantly Copy TB-Size Datasets](https://neon.com/blog/instantly-copy-tb-size-datasets-the-magic-of-copy-on-write), [Neon storage docs](https://neon.com/storage)

**PlanetScale** — Vitess-based; branches are literally separate MySQL instances seeded by copying schema (and optionally data via backup-restore, "Data Branching"), with deploy requests generating a semantic schema diff (via Vitess's `schemadiff`) and going through a serialized deploy queue for safe migration. This is *not* CoW storage-level branching — it's closer to "spin up a new instance and diff schemas," which is heavier (minutes, not seconds) but conceptually simpler and matches what a single-process platform could do with SQLite files directly. [PlanetScale branching docs](https://planetscale.com/docs/vitess/schema-changes/branching), [Database branching: three-way merge for schema changes](https://planetscale.com/blog/database-branching-three-way-merge-schema-changes)

**Turso / libSQL** — libSQL (SQLite fork, prod-ready) added CoW branching directly on SQLite semantics — branch a database instantly, test migrations, promote to production. This is the closest existing prior art to "SQLite-native git-style branching" and validates that the concept works at SQLite's scale/format. Turso's newer from-scratch Rust engine ("Limbo," now renamed Turso Database) is still alpha and not the thing to depend on; libSQL is the usable artifact. Build cost: low-to-medium if we vendor libSQL or its branching approach instead of stock SQLite. [libSQL](https://docs.turso.tech/libsql), [tursodatabase/libsql](https://github.com/tursodatabase/libsql)

**sqlite3session (SQLite Session Extension)** — Built into the SQLite amalgamation (since 3.13, disabled by default): attach a session object to a set of tables, and it records every INSERT/UPDATE/DELETE as an in-memory changeset; `sqlite3session_changeset()`/`patchset()` serialize it to a byte blob that can be applied to another SQLite DB with the same schema — literally "git diff/patch for a SQLite database," including 3-way merge support via changegroups. Maturity: rock solid, official SQLite C API, minimal footprint. Build cost: **low** — this is the single highest-leverage piece of prior art for this project. A daily/per-commit changeset file per app.db is small, diffable, storable as a git blob, and directly gives per-PR data branches (apply the base changeset stack, then the PR's incremental changeset) and time-travel restore (replay changesets up to a point, or store patchsets as inverse-diffs for rollback). [SQLite Session Extension intro](https://sqlite.org/sessionintro.html), [sqlite3session_changeset](https://www.sqlite.org/session/sqlite3session_changeset.html)

**lakeFS** — Git-like versioning over an object store using a 2-layer Merkle tree ("Range" leaf nodes + "Meta Range" index), giving CoW branches over S3-compatible blobs with high content reuse across commits that touch similar key prefixes. Maturity: production, used widely in MLOps. Build cost: medium to reimplement narrowly (a Merkle-indexed manifest over a blob dir), but it's solving multi-petabyte/billion-object problems this platform doesn't have — worth stealing the *manifest-of-content-hashes* idea, not the whole system. [lakeFS versioning internals](https://docs.lakefs.io/v1.73/understand/how/versioning-internals/)

**Xet (Hugging Face)** — Replaces Git LFS with content-defined chunking (~64KB chunks via rolling hash, so insertions/deletions don't invalidate the whole file's dedup) plus local/cached/global dedup lookups before upload; small edits to huge files (e.g. model checkpoints) upload only changed chunks. Maturity: production at HF scale. Build cost: medium — CDC chunking algorithms (FastCDC, Rabin fingerprinting) are well-documented and implementable in a weekend in TS; the global-dedup index is the harder infra piece. [Xet chunk-level deduplication spec](https://huggingface.co/docs/xet/en/deduplication)

**git-annex / DVC** — Simpler baseline: hash the file, move content to a content-addressed store (`.git/annex/data` or a DVC cache dir keyed by hash), replace the working-tree copy with a symlink/pointer, and let git version only the pointer/hash. No chunking — file-level dedup only. Maturity: both mature, widely used, but noticeably worse UX than chunked systems for large binary churn. Build cost: trivial — this is arguably the whole blob-storage design already, minus chunking. [git-annex for data management](https://scicomp.aalto.fi/scicomp/git-annex/)

### The 2–3 mechanisms to actually adopt

1. **sqlite3session changesets, git-committed** → per-PR data branches + time-travel restore. Low build cost (it's a C API SQLite already ships); wrap it in a thin TS binding. Store changesets as git blobs alongside app source so a PR branch literally carries its own data diff, and replay-to-timestamp gives restore for free.
2. **OS-level CoW clone (reflink/APFS clonefile) of the whole `app.db` file** → cheap Neon/Turso-style instant branch snapshots when a full-file copy is simpler than session-replay (e.g. forking a daughter platform's entire data dir). Medium cost, filesystem-dependent, needs a non-CoW fallback (plain copy) for portability.
3. **Content-defined chunking (FastCDC) for the blob/files side only** → git alone chokes on large-binary churn; chunking gives Xet/git-annex-style dedup for the "files" half of "app.db + files" without needing lakeFS's whole Merkle-manifest apparatus. Medium build cost, isolated to the blob subsystem.

Surprise: SQLite already ships an official, mature, git-diff-like changeset mechanism (sqlite3session) that is almost purpose-built for exactly the "data tied to source control" idea — this de-risks the whole pivot considerably more than expected. Gap: no prior art found for combining *both* structured-data changesets and chunked blob dedup under one unified git-native versioning scheme for a single-process app platform — that combination looks like genuinely novel territory, not a reimplementation of an existing system.

## Epic Games' Lore — Research Findings

**What it is (confirmed):** Lore is a next-generation, open-source **version control system** (not a database, not a blob store) built by Epic Games, publicly released ~June 2026. It's the successor to "Unreal Revision Control," the built-in VCS for UEFN (Unreal Editor for Fortnite). It targets projects that mix source code with large binary assets (game textures, meshes, audio) — same territory as Perforce/git-LFS, but rebuilt from scratch as "binary-first by design" rather than text-first-with-an-add-on.

**Repo/maturity (confirmed):** github.com/EpicGames/lore. Rust (81.7%), MIT license, v0.8.4 (pre-1.0, released June 26 2026), 7.7k stars / 340 forks / 44 open issues. Explicitly "pre-1.0... interfaces, on-disk formats, and APIs may change," though data written is meant to stay forward-readable. Separate SDK repos: lore-dotnet, lore-js, lore-python, lore-go — plus native C/C++ and Rust bindings. This is a real, actively-maintained, freshly-launched project with genuine Epic internal production use (UEFN), not a toy.

**Usable as library or only product (confirmed):** Both. It ships as a standalone CLI/server (local "demo mode" needing zero infra, or networked server deployment) **and** as an embeddable library with first-class bindings in C/C++, Rust, C#, Go, Python, JS. The docs advertise "one-to-one CLI access to full functionality" plus a "full-surface API" across languages — meaning you could vendor it as a data layer rather than shell out to a CLI.

**Architecture (confirmed, from the system-design doc):**
- **Content-addressed storage** via BLAKE3 hashing. Two subsystems: an *immutable store* (content keyed by hash — write once, verify on read) and a *mutable store* (branch pointers, name→ID mappings, lock state) with compare-and-swap atomicity.
- **Merkle trees**: file-node hash = content hash; directory-node hash = sorted hash-list of children; root hash = whole-repo state. Two revisions differing in one file share every unchanged subtree automatically — dedup is structural, not a heuristic.
- **Revision chain**: a DAG, each revision cryptographically linking to parent(s) (1 for commits, 2 for merges). Branches are mutable *pointers* to revisions, not containers — cheap to create/archive without losing history ("identity vs name" decoupling).
- **Chunking**: two strategies, chosen per content type by the calling application (not by Lore itself) — FastCDC content-defined chunking (~64KiB avg, 32KiB floor/256KiB ceiling rolling-hash boundaries, so edits inside a multi-GB file only re-chunk the touched region) vs fixed-size chunking (trivial, offset-addressable, better for random-access reads at the cost of dedup after insertions). "Recursive fragmentation" turns oversized fragment lists into fragment trees themselves, enabling true sparse byte-range reads without materializing the whole file.
- **Storage backend**: reference impl is on-disk append-only packfiles with mmappable indexes locally, AWS S3 + DynamoDB for the networked server — but storage/transport sit behind documented interfaces, explicitly designed for third-party backends (hot/warm/cold tiering, edge read replicas) "without forking the codebase."
- **Wire protocol**: fragments transferred individually and in parallel across connections/sessions, independent of each other ("reading fragment X never requires knowing about fragment Y"), multiplexed/pipelined sessions batching thousands of ops per call. Wire format is the versioned spec; on-disk format is implementation-specific.
- **Locking vs merging**: hybrid, not pure-lock and not pure-merge. Text gets standard three-way merge. Binaries get advisory (not enforced) locks — "informs rather than enforces" — with an explicit next-gen goal of scaling locks across branches at millions-of-files/thousands-of-users scale. Unresolvable binary conflicts surface as explicit divergence for the user to pick a winner; "pluggable resolvers by content type" is roadmap, not shipped.
- **Partial checkout/virtualization**: "sparse by construction" — clients materialize only what's requested and fetch fragments on demand. View files control inbound filtering (what to hydrate); ignore files control outbound filtering (what to stage). This is the FastCDC + Merkle-tree + fragment-cache combo doing git-sparse-checkout-plus-LFS-plus-VFS-for-Git in one coherent model instead of three bolted-together tools.
- **Access model**: partition-scoped (repo-level), no per-path ACLs inside a partition — compose finer-grained access via links between partitions instead.

**Ideas worth stealing for a code+data-versioned platform:**
1. Merkle-tree-of-hashes as the *only* identity model — trivially gives you diffing, dedup, and "same content across branches/PRs shares storage" for free, which is exactly what per-PR data branches / time-travel restore need.
2. Chunking strategy as a pluggable policy per content-type (CDC for logs/exports where dedup matters, fixed-size for random-access blobs like SQLite files) — directly relevant if app.db files are chunked for storage.
3. Sparse/on-demand hydration via view+ignore files — a much cleaner mental model than git-lfs's smudge/clean filters for "checkout only the data this daughter/branch needs."
4. Decoupling identity (content hash) from name (branch pointer) — cheap branch-per-PR data snapshots without duplicating storage, which maps well onto "seedable data" and per-PR data branches.
5. Explicit divergence over silent merge for binaries (SQLite files are binary) — advisory locks + explicit conflict surfacing is more honest than trying to 3-way-merge a .db file.

**Gaps/caveats:** Pre-1.0, API/format churn expected; reference server backend (S3+DynamoDB) is heavier infra than a single-Bun-process platform wants — you'd be reimplementing the ideas, not embedding Lore itself, unless you accept a Rust dependency plus its S3-shaped assumptions. No public production case studies beyond Epic/UEFN found. Didn't find independent benchmarks vs Perforce/git-LFS at scale — all performance claims are Epic's own docs.

## SQLite-as-primary-db research: dense findings

**1. Single-writer/WAL concurrency reality**
WAL mode still permits exactly one writer at a time — concurrent write transactions queue or `SQLITE_BUSY`. Real numbers: Forward Email production reports ~10,548 inserts/sec, 17,494 selects/sec on Node 20 (Apr 2026); general guidance is 10k–50k writes/sec ceiling on modern NVMe, 100k+ reads/sec. PocketBase field reports: "tens of thousands of concurrent reads, thousands of concurrent writes/sec" on a $6/mo 2vCPU/4GB Hetzner box serving 10k+ persistent connections — but explicitly "write-heavy workloads struggle, horizontal scaling impossible." Breakage pattern for your architecture: single-tenant-per-app-db sidesteps the worst of this (each app's writer contention is isolated to its own file), so the ceiling that matters is per-app request rate, not aggregate platform throughput — a normal CRUD app is nowhere near 10k writes/sec. Real risk is **bursty co-located writes** (e.g., a migration job + user traffic on the same app.db) and **long-held write transactions blocking readers momentarily** during checkpoint. SQLite's `BEGIN CONCURRENT` (multi-writer) exists only in an experimental branch, not upstream.

**2. bun:sqlite**
Native, no bindings; ~3–6x faster than better-sqlite3, 8–9x faster than Deno's driver for reads. Synchronous API, prepared statements, named/positional params, BLOB↔Uint8Array, class-mapped rows, bigint support. Inherits SQLite's own limits (281TB max, but practically great to ~100GB). Bun's own docs position it as fine for "single-server apps, desktop apps, microservices" and explicitly punt to Postgres/MySQL for "distributed systems requiring multiple write nodes" — which validates the per-app-single-writer model but confirms Bun adds no multi-writer capability of its own.

**3. Durability/replication tooling**
- **Litestream** (Ben Johnson): actively maintained in 2026, recent commits (write control, auth improvements). However there's an **open Feb 2026 issue reporting database corruption on restore** in production — no formal restore SLA/guarantee is documented. Treat as "battle-tested but not bulletproof," verify restores regularly.
- **LiteFS**: not archived, last push May 2026, "stable in production" per Fly docs but explicitly **pre-1.0, APIs may change**. Note: **LiteFS Cloud (managed control plane) was sunset Oct 2024** — only the self-hosted OSS project remains supported. Fly itself is de-emphasizing it relative to their newer Postgres offerings.
- **Turso/libSQL**: libSQL is a maintained, open-source SQLite fork (C, production-ready) adding server mode, HTTP replication, embedded replicas (in-process local SQLite synced from remote primary — microsecond local reads, consistency w/ primary). Turso's separate "Turso Database" (née Limbo) is a clean-room Rust rewrite, **still in beta**, not the thing running production today. Turso also ships experimental multi-writer support ("Beyond the Single-Writer Limitation") on their forked engine — a divergence from stock SQLite, so tightly coupled to their fork if adopted.

**4. Feature gaps vs Postgres**
No stored procedures/UDFs in the enterprise sense; no native multi-writer; smaller extension ecosystem. Answers: **FTS5** (full-text, mature, in-core), **JSON1** (mature, in-core), **sqlite-vec** (successor to sqlite-vss, active, embeddable vector search — not pgvector-scale but adequate for per-app RAG/embedding use cases without an external service). Geospatial via SpatiaLite exists but is heavier-weight than PostGIS. For "one small business app," these gaps rarely bite; they bite at analytics/reporting scale or genuine multi-writer concurrency needs.

**5. Prior art**
- **val.town**: literally "every val/project gets its own SQLite DB," backed by Turso libSQL; **forking a project optionally forks/copies its database** — this is the closest existing precedent for your per-app-data-dir + git-branch idea. Free tier 10MB, paid 1GB per db — i.e., they treat SQLite-per-tenant as small-data-only, not a general blob+DB store.
- **Cloudflare D1/Durable Objects**: SQLite-backed per-object storage is now core Cloudflare infra (Storage Relay Service powers both). Object-level isolation (1 tenant = 1 SQLite = 1 DO) is the same shape you're proposing. Storage capped 1GB→10GB per object at GA — signals Cloudflare doesn't trust SQLite-per-tenant past low-GB scale either.
- **rqlite**: Raft-replicated SQLite; writes 10–hundreds/sec depending on disk/network, writes blocked during snapshot creation, explicitly "not for high-write workloads." Relevant only if you wanted multi-node HA per app — probably overkill for your model.
- **PocketBase**: single SQLite file, hobby-project-with-no-company-backing per its own creator's warning, "not recommended for production critical apps." Useful cautionary data point given your platform is meant to be a serious hosting substrate.

**6. Snapshot/branching mechanics**
No direct hit on "VACUUM INTO + APFS clonefile" combined, but the pieces compose: APFS `clonefile`/`cp -c` is metadata-only CoW (proven near-instant regardless of file size, diverges only on write — this is Time Machine's mechanism). WAL mode means the *main db file* is only mutated at checkpoint; between checkpoints it's stable, so a clone of `app.db` while WAL is unmerged risks capturing a stale/inconsistent view unless you also clone the `-wal`/`-shm` files or force a full checkpoint (`PRAGMA wal_checkpoint(TRUNCATE)`) first. `VACUUM INTO 'file'` gives you a guaranteed-consistent full copy but is a **read+rewrite of the whole DB** (not CoW, not fast at 5GB — likely seconds, not sub-second). **Sub-1s CoW clone of a 5GB app.db is plausible on APFS/XFS reflink only if you clonefile the checkpointed main file directly (bypassing VACUUM INTO)** — this is unverified/untested in the sources found and should be prototyped, not assumed.

**Gaps/surprises**: no direct source confirms sub-second clonefile timing at multi-GB scale for SQLite files specifically (general APFS clonefile is metadata-only per docs, but real-world benchmarks for large DB files weren't found — worth a hands-on `cp -c` timing test on your target filesystem). Litestream's restore-corruption issue is a genuine yellow flag for a bet-the-platform decision. Turso's own multi-writer feature is fork-specific — don't conflate it with stock SQLite behavior if evaluating "does bun:sqlite get this too" (it doesn't).

## Attack

**1. Concrete failure cases for real client apps.**
- *50 concurrent users + background writer:* WAL gives one writer. A nightly import or migration holding a write transaction turns every user write into `SQLITE_BUSY`/queueing; with bun:sqlite's **synchronous** API, a busy-wait blocks the app's event loop itself — the whole Bun process stalls, not just the query. Postgres degrades a query; SQLite-in-process degrades the server.
- *Reporting query on 20GB:* a long analytical read in WAL mode pins the WAL (checkpoint starvation) — the `-wal` file grows unboundedly while the report runs, ballooning disk and slowing all writes. On Postgres this is a boring case. Val.town caps at 1GB, Cloudflare D1 at 10GB — the two closest prior-art operators both refuse to run SQLite-per-tenant at 20GB. You'd be operating past the envelope everyone else fenced off.
- *Anything needing LISTEN/NOTIFY, advisory locks, connection-level auth, stored procedures, PostGIS:* not "run your own container" edge cases for agencies — better-auth, Payload, Directus, most Rails/Django SOW stacks default to Postgres. "Bring your own DB container" means those apps lose ALL platform data services (backup, branching, quotas) exactly when their data matters most — the primitive silently excludes the serious tenants.

**2. The "Supabase third" collapses.** Supabase's pitch is not "a database exists"; it's Postgres + RLS + realtime + auth + storage with presigned URLs + a SQL editor over a *server* clients can also hit directly (BI tools, `psql`, ETL, Metabase). SQLite is in-process: **no network endpoint**, so no external BI tool, no ORM studio connecting from a laptop, no cron job in another container touching the data without a bespoke RPC layer you'd have to build (at which point you've rebuilt a database server, badly). Calling app.db+files "Supabase-like" is marketing debt: the moment a client asks "can I connect Metabase?" the answer is no. RBAC-7's role/RLS path also dies — SQLite has no roles.

**3. Multi-container/replica wall is structural, not tunable.** Any app with a web container + worker container (the *default* shape for anything with background jobs) needs both processes writing one file across container boundaries — SQLite over shared volumes with two containers is exactly the documented corruption scenario (POSIX locks across mount namespaces are shaky), and horizontal scaling (2 replicas for zero-downtime deploy!) is impossible. Note: even **blue-green deploys** break — old and new app versions briefly co-writing app.db is a two-writer scenario. The single-writer wall isn't a scale ceiling; it hits at replica count 2, which is deploy hygiene, not scale.

**4. Backup integrity across app.db + files is the weakest technical link.** A crash-consistent snapshot of a *directory* while an app writes requires either (a) filesystem-level atomic snapshot of db+wal+shm+files together — needs ZFS/btrfs/LVM, unavailable on arbitrary VPSes and inside overlayfs Docker volumes ("cp -c on APFS" is a dev-laptop fact, not a prod fact; typical target is ext4, **no reflink**); or (b) app cooperation (checkpoint + write-fence), which the platform can't force on a container it doesn't control. Worse: app.db and files/ can be *mutually* inconsistent (row committed referencing a file not yet fsynced) — no snapshot mechanism fixes application-level cross-store atomicity that Postgres+S3 also lacked, but Postgres at least had a single-store transactional core. And the designated off-host tool, Litestream, has an **open 2026 restore-corruption issue** and no restore SLA — and Litestream covers only app.db, not files/, so you need a second mechanism anyway and the "one data primitive" claim is already false at the backup layer.

**5. Deleting the S3 subset breaks the ecosystem contract.** Presigned URLs are the load-bearing feature: direct browser upload/download bypassing the app (large files not proxied through Bun), time-limited public sharing, and compatibility with every SDK/framework plugin (Uppy, next-s3-upload, Django-storages, image CDNs) that speaks S3. "files/ on a volume" forces all bytes through the app process, gives no cross-app or external access story, and orphans any client app already coded against the S3 SDK. The S3 subset was ~days of work and buys an entire ecosystem; deleting it saves almost nothing.

**6. Verdict: SQLite-first survives, narrowly — as the *default*, not the *only* primitive.** The mitosis story, sessions-changeset branching, and CoW seeding are genuinely stronger on SQLite, and 90% of SOW apps fit under the wall. But three mitigations are non-negotiable:

1. **Keep the S3-subset blob server** (it can *store* into the same per-app data dir and be versioned with it) — presigned URLs and SDK compat are the API; the fs layout is the implementation. Do not conflate deleting Postgres with deleting S3.
2. **Snapshot protocol, not snapshot hope:** backups/branches must run `PRAGMA wal_checkpoint(TRUNCATE)` via a platform sidecar or require apps to expose a quiesce hook; fallback path (`VACUUM INTO` + rsync of files/) must exist for ext4/no-reflink hosts, and restores must be **continuously verified** (integrity_check in CI) given Litestream's record.
3. **Postgres as a supported, documented escape hatch with platform backups** — a recipe (compose fragment + pg_dump into the same data-dir backup pipeline), so serious apps don't exit the platform's durability guarantees. Cap the advertised envelope honestly: SQLite tier ≤ ~10GB, single-writer, single-replica.
